⚡ Edge-cache speed demo

Read the same object three ways — raw S3, the Source Cooperative proxy, and the PR #189 chunk-cache preview — straight from your browser, and watch the difference per access pattern.

  • Cold = first request — a cache MISS that also populates the cache. Warm = median of the remaining requests, once the edge cache is filled.
  • Latency (ms — TTFB, footer, DuckDB query time): ↓ lower is better. Throughput (MB/s): ↑ higher is better. Each result states its direction, and the fastest endpoint's row is highlighted.
  • x-cache = the proxy's disposition for that request: HIT served from the edge cache · MISS fetched from origin, now cached · BYPASS deliberately not cached (full-object, conditional, > 32 MiB) · — no proxy at all (direct S3).
  • The comparison summary under each result pits the edge-cache proxy against both the no-cache proxy and direct S3. Absolute numbers swing with your network — judge the ratios and the x-cache disposition, and re-run a few times.

Methodology: all requests use cache: no-store so the browser cache never hides the network, and no cache-busting query param is added (the proxy only chunk-caches empty-query requests). Requests are interleaved across endpoints so network drift hits them evenly.
